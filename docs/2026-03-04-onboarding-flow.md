- `.claude/settings.json` for Claude Code

## Best Practices → agenTerm Features

### What we can directly build into the system:

#### 1. Auto-generate CLAUDE.md / AGENTS.md per worktree

**Best practice:** Every worktree should have a context file that tells the agent about the project, conventions, and current task scope.

**agenTerm integration:**
When agenTerm creates a worktree for a task, it auto-generates a context file containing:
- Project-level conventions (from a user-defined template stored in the project)
- Task-specific instructions (from the planning session output)
- Completion criteria (what "done" means for this task)
- Behavioral rules: "Do not ask for user input. If uncertain, write your question to BLOCKED.md and stop."

```markdown
# CLAUDE.md (auto-generated by agenTerm)

## Project
E-commerce backend API (Go + PostgreSQL)

## Task
Implement JWT refresh token rotation in /auth/tokens.go

## Completion Criteria
- All existing tests pass
- New tests cover: token rotation, expiry, concurrent refresh
- Run `go test ./auth/...` before marking done
- Commit with message prefix: [agenterm:task-042]

## Rules
- Do NOT ask for user input. You are running autonomously.
- If you encounter an ambiguous decision, write it to BLOCKED.md and stop.
- Run tests after every significant change.
- Commit working code frequently with descriptive messages.

## Conventions
- Error handling: always wrap with fmt.Errorf("context: %w", err)
- Tests: table-driven tests, use testify/assert
- No global state
```

**Applicable to:** All agents (Claude Code reads CLAUDE.md, Codex reads AGENTS.md, OpenCode reads AGENTS.md, Kimi reads AGENTS.md)

#### 2. Auto-compact / context management

**Best practice:** Claude Code sessions degrade when context grows too large. Regular `/compact` keeps quality high.

**agenTerm integration:**
- Monitor TUI output for token usage indicators (Claude Code shows token count in status bar)
- The output parser can detect when context is getting large
- Inject `/compact` command into the session automatically at configurable thresholds
- Alternative: include a rule in CLAUDE.md: "Run /compact when you notice performance degradation or after completing a sub-task"

**Applicable to:** Claude Code (native `/compact`), other agents may have equivalents (`/compact` in Kimi CLI, context compaction in OpenCode)

#### 3. Git worktree lifecycle management

**Best practice:** One worktree per task, clean branch, isolated from other work.

**agenTerm integration (already partially built):**
- Planning session outputs a worktree structure with parallel/sequential relationships
- "One-click create" generates all worktrees from a single action:
  - `git worktree add .agenterm/worktrees/<task-id> -b <branch-name>`
  - Copies `.worktreeinclude` files (like `.env`) if configured
  - Writes the task-specific CLAUDE.md/AGENTS.md into the worktree
  - Writes the permission config files into the worktree
- On task completion + merge, auto-cleanup:
  - `git worktree remove .agenterm/worktrees/<task-id>`
  - `git branch -d <branch-name>`

**Applicable to:** All agents (worktree is a git feature, agent-agnostic)

#### 4. Commit conventions for task tracking

**Best practice:** Use commit messages as context checkpoints, not just code history.

**agenTerm integration:**
- Enforce commit message prefix via CLAUDE.md/AGENTS.md: `[agenterm:task-XXX]`
- agenTerm's auto-commit hook (already exists) can add task metadata
- On task completion, agenTerm parses `git log` from the worktree to auto-generate a changelog for the verification dashboard
- Detect `[READY_FOR_REVIEW]` in commits to trigger review stage (already built)
- Detect `[BLOCKED]` in commits to mark task as blocked

**Applicable to:** All agents

#### 5. Plan mode before build

**Best practice:** Use plan mode (`/plan` or `Shift+Tab`) to think through architecture before writing code.

**agenTerm integration:**
- The planning session with the planner agent IS this step, externalized
- For agents with native plan mode (Claude Code), include in CLAUDE.md: "Start by creating a plan. Review the task requirements and list the files you will modify before writing any code."
- For agents without plan mode, the task description from the planning session serves as the pre-made plan

**Applicable to:** All agents (as a prompt instruction)

#### 6. TDD workflow integration

**Best practice:** Write tests first, then implement, then refactor.

**agenTerm integration:**
- Already have TDD playbook concept
- Include in CLAUDE.md when TDD mode is selected: "Follow strict TDD: write a failing test first, implement minimal code to pass, then refactor. Run tests after each step."
- agenTerm can verify TDD compliance by checking git log: test commits should appear before implementation commits

**Applicable to:** All agents (as a prompt instruction)

#### 7. `/init` equivalent for workspace setup

**Best practice:** Run `/init` in a new workspace to let the agent understand the project.

**agenTerm integration:**
- On first project creation, run a one-time "project analysis" task:
  - Start a planner agent session
  - Agent reads the codebase and generates a project summary
  - Output is stored as "project knowledge" (already exists in agenTerm)
  - This knowledge is injected into every future task's CLAUDE.md
- Essentially, agenTerm does `/init` once and caches the result for all future sessions

**Applicable to:** All agents (the output feeds into CLAUDE.md/AGENTS.md)

#### 8. Session naming and resume

**Best practice:** Name sessions descriptively to find them later.

**agenTerm integration:**
- agenTerm already names sessions by task ID and agent
- Session output is stored in the database
- If an agent session crashes, agenTerm can restart it in the same worktree — the CLAUDE.md provides the context to resume

**Applicable to:** Claude Code (native `/resume`), others restart with context from CLAUDE.md

## What stays as "recommendations" (not built-in):

These are workflow tips shown during onboarding but not enforced by the system:

1. **Keep worktrees to 3-4 active** — more than that reduces focus and increases merge complexity
2. **Review diffs, not full code** — focus verification on what changed, trust the agent for boilerplate
3. **Start with simple tasks** — validate the autonomous workflow with well-scoped work before tackling complex features
4. **Iterate on CLAUDE.md** — after each cycle, update the project template with things the agent got wrong
5. **Use focused models** — assign cheaper/faster models to simpler tasks (linting, formatting), expensive models to complex tasks (architecture, debugging)

## Summary: Onboarding Generates Everything

After the 3-step onboarding, a new user has:

| Asset | Generated by |
|-------|-------------|
| Agent registry (YAML per agent) | Step 1 |
| Permission templates per agent | Step 2 |
| CLAUDE.md / AGENTS.md project template | Step 3 (auto + editable) |
| Project knowledge (codebase summary) | Auto `/init` on first project |
| Worktree creation scripts | Built into the system |
| Commit conventions | Embedded in CLAUDE.md template |
| Auto-compact rules | Embedded in CLAUDE.md template |

The user's only ongoing work: define requirements → planning session → click "launch" → come back and test.
